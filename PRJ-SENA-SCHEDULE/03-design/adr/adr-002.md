# ADR-002: ORM vs SQL Raw para PostgreSQL

**Status:** Aprobado ✅  
**Fecha:** 2026-05-20  
**Decisor:** A06 (Tech Lead)  
**Stakeholders:** A07 (Backend Dev), A10 (Data Architect), A13 (DBA)

---

## Contexto

Backend Go necesita persistencia en PostgreSQL para 5 tablas (ambientes, instructores, fichas, horarios, observaciones) con relaciones 1:N y N:1. Decisión crítica: usar ORM (GORM) o SQL raw con driver pgx.

**Requisitos:**
- Queries complejas con JOINs (calendario ficha, agenda instructor)
- Validaciones de conflictos con overlapping time ranges
- Transacciones ACID (crear horario = insert + validar + commit)
- Performance: queries <200ms, validaciones <500ms
- Mantenibilidad: equipo pequeño (2 devs), curva de aprendizaje baja

**Trade-offs a evaluar:**
- Productividad vs Control
- Abstracción vs Performance
- Type safety vs Flexibilidad

---

## Opciones Consideradas

### Opción 1: GORM (ORM)

**Descripción:** ORM completo para Go con migraciones, associations, hooks, soft deletes.

**Ventajas:**
- ✅ Menos código: `db.Create(&horario)` vs 5 líneas SQL
- ✅ Type-safe: compilador detecta errores en struct tags
- ✅ Migraciones automáticas: `db.AutoMigrate(&Horario{})` (desarrollo rápido)
- ✅ Soft deletes nativos: `deleted_at` sin código adicional
- ✅ Associations: `db.Preload("Ambiente").Find(&horarios)` carga relaciones automáticamente

**Desventajas:**
- ❌ **Queries complejas limitadas:** Validación de conflictos (time range overlap) requiere SQL raw de todos modos
- ❌ **Performance overhead:** N+1 queries en preloads (mitigable con `Preload` pero agrega complejidad)
- ❌ **"Magia" oculta:** `AutoMigrate` puede generar esquemas subóptimos (indexes faltantes)
- ❌ **Debugging difícil:** Errores de SQL generado no son obvios
- ❌ **Control de schema limitado:** Constraints complejos (CHECK, exclusion) requieren SQL raw

**Ejemplo código:**
```go
// GORM
type Horario struct {
    ID           uint
    Fecha        time.Time
    HoraInicio   time.Time
    HoraFin      time.Time
    AmbienteID   uint
    Ambiente     Ambiente
    InstructorID uint
    Instructor   Instructor
}

db.Create(&horario) // INSERT automático

// Problema: Validación de conflictos requiere SQL raw de todos modos
db.Raw(`SELECT * FROM horarios WHERE ambiente_id = ? AND ...`, ambienteID).Scan(&existentes)
```

### Opción 2: pgx (SQL Raw)

**Descripción:** Driver PostgreSQL nativo con performance óptima, sin abstracción ORM.

**Ventajas:**
- ✅ **Control total:** Queries exactas sin "magia" — performance predecible
- ✅ **Performance máxima:** pgx es el driver más rápido para PostgreSQL (2x más rápido que pq)
- ✅ **Queries complejas naturales:** Time range overlaps, window functions, CTEs sin limitaciones
- ✅ **Debugging simple:** SQL visible en código — errores obvios
- ✅ **Schema explícito:** Migraciones con golang-migrate (archivos `.sql` versionados) — control total
- ✅ **Constraints complejos:** `EXCLUDE USING gist` para time overlaps directo en BD

**Desventajas:**
- ⚠️ **Más boilerplate:** `db.Query()` + `rows.Scan()` vs `db.Find()`
- ⚠️ **No type-safe en queries:** `SELECT ambiente_id` con typo no detectado en compilación
- ⚠️ **Manejo manual de NULL:** `sql.NullString` vs punteros en GORM

**Ejemplo código:**
```go
// pgx
const query = `
    SELECT id, fecha, hora_inicio, hora_fin, ambiente_id, instructor_id
    FROM horarios
    WHERE ambiente_id = $1
      AND fecha = $2
      AND (hora_inicio, hora_fin) OVERLAPS ($3, $4)
`

rows, err := pool.Query(ctx, query, ambienteID, fecha, horaInicio, horaFin)
defer rows.Close()

var horarios []*Horario
for rows.Next() {
    h := &Horario{}
    err := rows.Scan(&h.ID, &h.Fecha, &h.HoraInicio, &h.HoraFin, &h.AmbienteID, &h.InstructorID)
    horarios = append(horarios, h)
}
```

### Opción 3: Híbrido (GORM para CRUD, Raw para complejos)

**Descripción:** Usar GORM para operaciones simples (Create, FindByID) y SQL raw para queries complejas.

**Ventajas:**
- ✅ Balance entre productividad y control
- ✅ CRUD rápido con GORM, queries complejas con raw SQL

**Desventajas:**
- ❌ **Inconsistencia:** Dos formas de hacer lo mismo — confusión en codebase
- ❌ **Doble mantenimiento:** Aprender GORM + SQL raw
- ❌ **Overhead de GORM** presente de todos modos (imports, configuración)
- ❌ **Testing más complejo:** Mocks para GORM + pgx

---

## Decisión

**Elegido: pgx (SQL Raw) con golang-migrate**

**Razones:**

1. **Queries complejas son core del negocio:** Validaciones de conflicto (HU-017, HU-018, HU-019) son el valor principal del MVP — requieren time range overlaps, que GORM no soporta elegantemente.

2. **Performance crítica:** Objetivo <500ms para validaciones — pgx 2x más rápido que GORM en benchmarks, y elimina N+1 queries.

3. **Schema control necesario:** A13 (DBA) diseñará índices específicos (GiST, exclusion constraints) — `AutoMigrate` de GORM no genera esto.

4. **Simplicidad arquitectónica:** Hexagonal ya abstrae infraestructura — ORM agrega capa adicional sin valor (repositories ya son la abstracción).

5. **Curva de aprendizaje equiparable:** A07 ya conoce SQL — aprender pgx API (Query, Scan) toma <1 día vs aprender GORM (Preload, associations, hooks) toma 3-5 días.

6. **Debugging productivo:** SQL visible en código — errores de performance detectables con EXPLAIN sin "magia" oculta.

7. **Trade-off aceptable:** Más boilerplate (5-10 líneas extra por query) justificado por control total + performance + mantenibilidad.

## Consecuencias

### Positivas

- ✅ Queries explícitas — A09 puede optimizar con EXPLAIN directamente
- ✅ Migraciones como código SQL versionado — rollback trivial
- ✅ Testing con testcontainers — DB real sin mocks complejos
- ✅ Performance predecible — sin sorpresas de N+1 queries
- ✅ Constraints de BD nativos — validación a nivel DB (última defensa)

### Negativas

- ⚠️ Más líneas de código (20% más vs GORM en CRUDs simples)
- ⚠️ Scan manual propenso a errores (orden de columnas debe coincidir)
- ⚠️ No type-safe en queries (typos en nombres de columnas no detectados en compilación)

### Mitigación de Riesgos

- **Code generation:** Considerar sqlc (v1.1) para generar Go código desde SQL queries (type-safe + menos boilerplate)
- **Query helpers:** Crear funciones reutilizables `scanHorario()`, `scanAmbiente()` para reducir repetición
- **Tests de integración:** Cobertura ≥70% en repositories — detectar errores de Scan temprano

## Alternativas Rechazadas

| Opción | Razón de Rechazo |
|--------|------------------|
| **sqlx** | Menos features que pgx (no connection pooling nativo), comunidad más pequeña |
| **ent (Facebook)** | ORM muy complejo, overhead no justificado para 5 tablas |
| **sqlc** | Code generator interesante pero agrega paso de build — evaluable en v1.1 |
