# ADR-001: Framework HTTP para Backend Go

**Status:** Aprobado ✅  
**Fecha:** 2026-05-20  
**Decisor:** A06 (Tech Lead)  
**Stakeholders:** A07 (Backend Dev), A09 (QA)

---

## Contexto

El backend Go requiere un framework HTTP para exponer API REST con routing, middleware (CORS, logging, recovery), parseo de JSON, y manejo de errores. Necesitamos balance entre performance, simplicidad y productividad para equipo de 2 devs con timeline 14 semanas.

**Requisitos:**
- Routing con parámetros de ruta (`:id`)
- Middleware chain (CORS, logger, auth placeholder)
- JSON bind automático (request → struct)
- Validación de requests
- Performance ≥1000 req/s (benchmark estándar)
- Documentación y comunidad activa
- Curva de aprendizaje baja (equipo pequeño)

---

## Opciones Consideradas

### Opción 1: Gin (github.com/gin-gonic/gin)

**Ventajas:**
- ✅ Más rápido que net/http con router nativo (8x según benchmarks)
- ✅ Middleware chain muy intuitivo: `r.Use(Logger(), CORS())`
- ✅ JSON binding con validación integrada: `c.ShouldBindJSON(&req)`
- ✅ Documentación excelente + comunidad muy activa (77k stars GitHub)
- ✅ Usado en producción por empresas grandes (Alibaba, Tencent)
- ✅ Curva de aprendizaje suave (API similar a Express.js)

**Desventajas:**
- ⚠️ "Opinionated" — menos flexible que net/http raw
- ⚠️ Dependencia externa (riesgo de mantenimiento, pero activamente desarrollado)

**Ejemplo código:**
```go
r := gin.Default()
r.POST("/api/v1/horarios", horarioHandler.CrearHorario)
r.GET("/api/v1/horarios/:id", horarioHandler.ObtenerHorario)
r.Run(":8080")
```

---

### Opción 2: Chi (github.com/go-chi/chi)

**Ventajas:**
- ✅ Lightweight (3x más pequeño que Gin)
- ✅ 100% compatible con `net/http` (usa `http.Handler` estándar)
- ✅ Routing con middleware por grupo: `r.Route("/api/v1", func(r chi.Router) {...})`
- ✅ Sin "magia" — más transparente que Gin

**Desventajas:**
- ❌ No tiene JSON binding automático (más boilerplate)
- ❌ Validación de requests manual (requiere librería externa go-playground/validator)
- ❌ Menos middleware preconfigurado (logger, recovery no incluidos)
- ❌ Comunidad más pequeña que Gin (17k stars)

**Ejemplo código:**
```go
r := chi.NewRouter()
r.Use(middleware.Logger)
r.Post("/api/v1/horarios", horarioHandler.CrearHorario)
r.Get("/api/v1/horarios/{id}", horarioHandler.ObtenerHorario)
http.ListenAndServe(":8080", r)
```

---

### Opción 3: net/http (Standard Library)

**Ventajas:**
- ✅ Zero dependencias externas
- ✅ Máxima flexibilidad y control
- ✅ Mantenido por Go core team (estabilidad garantizada)

**Desventajas:**
- ❌ Routing manual con `mux.HandleFunc()` — sin parámetros de ruta nativos (`:id`)
- ❌ Middleware chain manual (mucho boilerplate)
- ❌ JSON parsing completamente manual: `json.NewDecoder(r.Body).Decode(&req)`
- ❌ Sin validación integrada
- ❌ Productividad baja (2-3x más código que Gin para mismo resultado)

**Ejemplo código:**
```go
mux := http.NewServeMux()
mux.HandleFunc("/api/v1/horarios", horarioHandler.CrearHorario)
http.ListenAndServe(":8080", mux)
// Parsing manual en handler:
var req CrearHorarioRequest
json.NewDecoder(r.Body).Decode(&req)
```

---

## Decisión

**Elegido: Gin (github.com/gin-gonic/gin)**

**Razones:**

1. **Productividad crítica:** Timeline 14 semanas requiere velocidad de desarrollo — Gin reduce boilerplate 50% vs net/http.

2. **Performance suficiente:** 1000+ req/s en benchmarks, muy superior a los 100 req/s objetivo MVP.

3. **JSON binding + validación integrados:** `c.ShouldBindJSON()` + tags de validación eliminan código repetitivo.

4. **Middleware ecosystem:** Logger, CORS, Recovery incluidos — net/http requeriría implementarlos manualmente.

5. **Curva de aprendizaje baja:** A07 (backend dev) puede ser productivo en <2 días con Gin vs 1 semana con Chi+validator.

6. **Comunidad y recursos:** Stack Overflow, tutoriales, ejemplos abundantes — reduce tiempo de troubleshooting.

7. **Trade-off aceptable:** Dependencia externa justificada por ganancia en productividad + estabilidad demostrada.

---

## Consecuencias

### Positivas

- ✅ Menos código a escribir y mantener (handlers más limpios)
- ✅ Validación de requests declarativa con tags: `json:"fecha" binding:"required"`
- ✅ Testing más simple con `httptest.NewRecorder()` + Gin context mock
- ✅ Error handling consistente: `c.JSON(400, gin.H{"error": "mensaje"})`

### Negativas

- ⚠️ Dependencia externa — si Gin se descontinúa, migración requerida (riesgo bajo: 77k stars, activamente mantenido)
- ⚠️ Menos control granular que net/http raw (trade-off aceptado)

### Mitigación de Riesgos

- Abstraer Gin detrás de ports/in (Hexagonal) — handlers implementan interfaces, no exponen Gin directamente
- Si migración futura requerida: adapters/in/http se reescriben, application/ y domain/ no se tocan (ventaja de Hexagonal)

---

## Alternativas Rechazadas

| Framework | Razón de Rechazo |
|-----------|------------------|
| **Fiber** | Demasiado "magia" (API similar a Express), menor madurez que Gin |
| **Echo** | Similar a Gin pero comunidad más pequeña, menos middleware preconfigurado |
| **Gorilla Mux** | Solo routing, sin JSON binding ni validación (requiere muchas librerías adicionales) |
